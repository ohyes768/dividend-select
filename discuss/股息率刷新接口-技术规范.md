# 股息率核心数据刷新接口 - 技术规范文档

## 1. 概述

### 1.1 背景

现有系统通过命令行工具 `main.py` 执行股息率数据刷新，支持完整运行模式和本地数据模式。为实现自动化数据更新（特别是通过 n8n 定时任务），需要提供一个 REST API 接口来执行相同的数据刷新操作。

### 1.2 目标

**核心目标（必达）：**
- 提供 `POST /api/dividend/refresh` 接口，用于刷新股息率核心数据
- 支持增量更新（断点续传），避免重复计算已处理的股票
- 返回处理统计信息，便于监控和验证

**附加目标（期望达成）：**
- 防止并发调用，避免数据冲突
- 提供清晰的错误处理和日志记录

**非目标（明确不做）：**
- 不更新板块映射数据（使用现有数据）
- 不移动历史数据到月度目录

### 1.3 目标用户群体

- n8n 定时任务调度系统（主要）
- 开发运维人员（调试和测试）

## 2. 功能需求

### 2.1 核心功能

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 获取股票列表 | 从红利指数持仓数据中获取符合条件的股票（基于分红次数阈值） | P0 |
| 计算股息率 | 对每只股票计算近 3 年股息率及相关数据 | P0 |
| 增量更新 | 检查已处理的股票，跳过并只处理新增或需要更新的股票 | P0 |
| 保存数据 | 将计算结果保存到 CSV 文件（当前日期目录） | P0 |
| 防止并发 | 拒绝并发刷新请求，返回 409 错误 | P1 |
| 参数配置 | 支持配置最小分红次数阈值 | P1 |

### 2.2 用户故事

**典型用户场景（n8n 定时任务）：**
1. n8n 在指定时间调用 `POST /api/dividend/refresh` 接口
2. 接口开始获取股票列表（使用默认或指定的分红次数阈值）
3. 检查已处理的股票，实现增量更新
4. 计算股息率并保存到数据文件
5. 返回处理统计信息（总数、新增数、文件路径、时间戳）
6. n8n 根据返回信息记录日志或触发后续操作

**开发调试场景：**
1. 开发者手动调用接口测试
2. 使用不同的分红次数阈值验证结果
3. 查看日志排查问题

## 3. 技术决策

### 3.1 技术栈

| 组件 | 选择 | 理由 |
|------|------|------|
| Web 框架 | FastAPI | 已在项目中使用，性能优秀，支持异步 |
| 数据计算 | DividendCalculator | 现有服务，已验证稳定 |
| 数据存储 | CSV 文件（data/YYYY-MM/） | 与现有实现保持一致 |
| 并发控制 | 全局标志位 | 简单有效，满足场景需求 |

### 3.2 架构设计

**系统架构（文字描述）：**
```
客户端 (n8n)
    ↓ POST /api/dividend/refresh
FastAPI 路由层 (routes.py)
    ↓ 检查并发状态
    ↓ 获取股票列表
    ↓ 过滤已处理股票
    ↓ 计算股息率
数据层 (DividendCalculator)
    ↓ API 调用获取历史数据
    ↓ 计算 3 年平均股息率
    ↓ 增量写入 CSV
存储层 (data/YYYY-MM/近3年股息率汇总.csv)
```

**模块划分：**
- `src/api/routes.py`: 新增刷新接口路由
- `src/api/models.py`: 新增请求/响应模型
- `src/core/dividend_calculator.py`: 复用现有计算服务

**数据流向：**
1. 接收请求参数 → 2. 检查并发标志 → 3. 设置并发标志 → 4. 获取股票列表 → 5. 过滤已处理股票 → 6. 计算股息率 → 7. 保存数据 → 8. 清除并发标志 → 9. 返回统计信息

**关键接口设计：**
```python
POST /api/dividend/refresh
Request Body: {
    "min_dividend": int  # 最小分红次数阈值，默认 5
}
Response: {
    "success": bool,
    "message": str,
    "stats": {
        "total_processed": int,      # 处理总数
        "new_or_updated": int,       # 新增/更新数
        "skipped": int,              # 跳过数（已存在）
        "file_path": str,            # 文件路径
        "start_time": str,           # 开始时间
        "end_time": str              # 结束时间
    }
}
```

### 3.3 方案权衡记录

#### 备选方案 1：完整刷新 vs 增量更新
| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 完整刷新 | 逻辑简单，无状态 | 耗时长，资源浪费 | ✅ 增量更新 |
| 增量更新 | 高效，支持断点续传 | 需要检查已处理股票 | 已选择 |

**最终选择：** 增量更新
**理由：** 股息率数据更新频率较低，大部分股票数据稳定，增量更新可大幅减少处理时间。

**潜在风险：** 如果需要强制刷新全部数据，需要额外支持。
**缓解措施：** 可通过参数控制是否使用增量模式（当前版本暂不实现）。

#### 备选方案 2：并发控制策略
| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 全局标志位 | 简单易实现 | 无法支持排队 | ✅ 全局标志位 |
| 任务队列 | 支持排队、限流 | 复杂度高，引入新依赖 | 未选择 |

**最终选择：** 全局标志位（模块级布尔变量）
**理由：** n8n 定时任务间隔较长（通常一天一次），并发概率极低，简单方案足够。

**潜在风险：** 如果确实需要并发支持，需要重构。
**缓解措施：** 返回清晰的错误信息（409 Conflict），调用方可延迟重试。

#### 备选方案 3：错误处理策略
| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 部分成功继续 | 容错性高，最大程度更新数据 | 需要记录错误详情 | ✅ 部分成功继续 |
| 全部或失败 | 数据一致性有保障 | 一次失败全部作废 | 未选择 |

**最终选择：** 部分成功继续
**理由：** 股息率数据计算独立，单只股票失败不应影响其他股票。

## 4. 数据设计

### 4.1 数据结构

**请求模型 (RefreshRequest)：**
```python
{
    "min_dividend": int  # 最小分红次数阈值，默认 5
}
```

**响应模型 (RefreshResponse)：**
```python
{
    "success": bool,     # 是否成功
    "message": str,      # 操作结果消息
    "stats": {
        "total_processed": int,      # 处理总数
        "new_or_updated": int,       # 新增/更新数
        "skipped": int,              # 跳过数（已存在）
        "file_path": str,            # 文件路径
        "start_time": str,           # 开始时间 (ISO 8601)
        "end_time": str              # 结束时间 (ISO 8601)
    }
}
```

### 4.2 状态管理

**全局并发标志：**
```python
# src/api/routes.py
_is_refreshing: bool = False
```

**状态流转规则：**
1. 收到请求 → 检查 `_is_refreshing`
2. 如果为 `True` → 返回 409 Conflict
3. 如果为 `False` → 设置为 `True`，开始处理
4. 处理完成（成功或失败） → 设置为 `False`
5. 异常情况 → 使用 try-finally 确保标志被清除

### 4.3 持久化策略

**数据文件：**
- 路径：`data/YYYY-MM/近3年股息率汇总.csv`
- 目录：根据当前日期自动生成（使用 `get_current_date_dir()`）
- 格式：与现有 CSV 格式一致

**增量写入：**
- 使用 `append_csv_row()` 函数追加写入
- 每处理完一只股票立即写入，避免内存堆积
- 文件头仅在首次写入时添加

## 7. 边缘情况与错误处理

### 7.1 异常场景

| 场景 | 处理策略 |
|------|----------|
| API 调用失败（akshare） | 记录错误日志，跳过当前股票，继续处理后续股票 |
| 文件写入失败 | 抛出异常，返回 500 错误 |
| 并发请求 | 返回 409 Conflict 错误，提示"刷新正在进行中" |
| 参数验证失败 | 返回 400 Bad Request 错误 |
| 数据目录不存在 | 自动创建目录 |

### 7.2 边界条件

| 边界条件 | 处理方式 |
|----------|----------|
| 股票列表为空 | 返回成功，`new_or_updated = 0` |
| 所有股票已处理 | 返回成功，`skipped = total_count` |
| `min_dividend = 0` | 验证为非法参数，返回 400 错误 |
| 分红次数阈值过大（如 > 100） | 正常处理，返回空列表 |

### 7.3 降级策略

- akshare API 超时：设置合理的超时时间，失败后继续处理
- 网络中断：记录错误，返回部分成功
- 服务崩溃：标志位可能未被清除，需要重启服务恢复

### 7.4 容错机制

```python
try:
    # 设置并发标志
    _is_refreshing = True
    start_time = datetime.now()

    # 执行刷新逻辑
    ...

    # 成功返回
    return {
        "success": True,
        "message": "刷新完成",
        "stats": {
            ...
        }
    }
except Exception as e:
    logger.error(f"刷新失败: {e}")
    raise HTTPException(status_code=500, detail=f"刷新失败: {str(e)}")
finally:
    # 确保标志被清除
    _is_refreshing = False
```
